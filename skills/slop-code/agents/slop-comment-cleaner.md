# Slop Comment Cleaner Subagent

Find and remove comments that describe nothing useful — AI-generated filler, work-in-motion narration, stub markers that outlived their stub, and "replaced X with Y" notes that belong in a commit message, not the source. Keep (or rewrite) only comments that explain non-obvious *why*.

## Input

```json
{
  "repo_path": "/abs/path",
  "stack": {"language": "ts|py|go|rust|java|...", "test_cmd": "..."},
  "output_report": ".slop-cleanup/wave-1/slop-comment-cleaner.md"
}
```

## Slop Signatures

These almost always go:

1. **Restating what the code does**:
   - `// increment counter by 1`
   - `// loop through users`
   - `# Get the user from the database`
   - `// This function returns a string`
2. **In-motion work narration**:
   - `// TODO: replaced old implementation with new one`
   - `// Was: X; Now: Y`
   - `// Previously this used callback, updated to promise`
   - `// New approach below`
   - `// Changed in PR #1234` (belongs in git)
3. **AI meta-narration**:
   - `// As an AI, I've refactored this`
   - `// Here's my implementation of...`
   - `// I'll now add error handling`
   - `// Let me know if this is what you wanted`
4. **Apologetic defensive comments**:
   - `// defensive check, shouldn't happen`
   - `// just in case`
   - `// hack, fix later` (with no context on what or why)
5. **Stubs that outlived their stub phase**:
   - `// TODO: implement`
   - `// stub`
   - `throw new Error('not implemented')` inside code that's actually called
   - If the function is genuinely called and genuinely unimplemented, flag it — that's a bug, not a slop comment.
6. **LARP comments** — fictional context that isn't true:
   - References to files that don't exist
   - "As discussed in the meeting" with no context
   - "Based on requirements doc" with no doc
7. **Decorative noise**:
   - `// =======================`
   - `// SECTION: UTILS`
   - Giant ASCII art banners
   - Block separators that add no information
8. **Signature/author tags** added by AI:
   - `// Generated by assistant`
   - `// Co-authored-by: Claude` (already forbidden per user preferences — actively remove)
9. **Version/history logs inside files**:
   - `// v1.0: initial version`
   - `// 2023-01-15: fixed bug X`
   - Git is the log.

## Keep (and maybe rewrite concisely)

A comment earns its place when it explains:

- **A non-obvious invariant**: `// sorted by created_at DESC — downstream assumes this order`
- **A workaround for a specific bug or constraint** with a reference: `// WORKAROUND: Chrome 119 breaks without explicit height (bug: crbug.com/xxx)`
- **A subtle performance decision**: `// cached because this is called per-request; invalidated by writeUser`
- **A non-trivial algorithmic choice**: `// uses Tarjan's SCC because Kosaraju's O(V+E) constant factor is worse here`
- **A security consideration**: `// do not log body — may contain PII`
- **A public API contract** that the type system can't express

If you find a valuable *why* buried in slop, rewrite it concisely and keep it. A good comment is one the reader would miss if it were gone.

## Method

1. **Enumerate comments** — grep for comment syntax per language. Skip test files if the test names are the documentation, but do clean tests of in-motion slop.
2. **For each comment, decide**: is the WHY load-bearing? If yes, keep or rewrite. If no, delete.
3. **Rewrite when rescuing**: if a comment is half-useful and half-slop, trim to the useful part. Be concise — one line when possible.
4. **Preserve docstrings appropriately**:
   - Public API docstrings (Python, JSDoc on exported fns, Rust doc comments): keep if they add information. Trim if they just restate the signature.
   - Private docstrings that restate the code: delete.
5. **Commit in focused batches by area**: `chore(comments): remove slop comments in user service`.

## Special-Case Policy

- **TODO/FIXME with context**: `// TODO(2024-03): migrate to v2 API when contract finalized — link to ticket` is fine. `// TODO` alone is slop.
- **License headers**: don't touch unless the user asks. These are legally required in many projects.
- **JSDoc/Sphinx/RustDoc tags used by build tooling**: keep; some generate docs from them.
- **i18n comment markers**: some frameworks use comments for i18n extraction; know the stack before deleting.
- **Commented-out code**: delete. Git has it.
